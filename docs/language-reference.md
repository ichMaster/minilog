## Синтаксис

### Формальна граматика (EBNF)

```
програма       = { інструкція } ;
інструкція     = факт | правило | запит ;

факт           = складений_терм "." ;

правило        = "правило" складений_терм "якщо" тіло "."
               | "правило" складений_терм ":" НОВИЙ_РЯДОК ВІДСТУП "якщо" тіло "." КІНЕЦЬ_ВІДСТУПУ ;

тіло           = ціль { "і" ціль } ;
ціль           = складений_терм | заперечення | порівняння ;
заперечення    = "не" складений_терм ;
порівняння     = терм оператор_порівняння терм ;
оператор_порівняння = "≥" | "≤" | ">" | "<" | "=" | "≠" ;

запит          = "?-" [ "слід" ] складений_терм "." ;

складений_терм = ІДЕНТ "(" [ терм { "," терм } ] ")"
               | ІДЕНТ ;

терм           = атом | змінна | число | рядок | складений_терм ;
атом           = ІДЕНТ ;
змінна         = "?" ІДЕНТ | "?_" ;
число          = ЦІЛЕ | ДРОБНЕ ;
рядок          = '"' { символ } '"' ;

ІДЕНТ          = (Літера | "_") { Літера | Цифра | "_" | "'" } ;
коментар       = "%" { символ } НОВИЙ_РЯДОК ;
```

## Терми

Терми — базові будівельні блоки minilog.

### Атоми

Іменовані константи. Починаються з літери або `_`.

```
авраам
ісак
червоний
stg_customers
```

### Змінні

Починаються з `?`. Анонімна змінна — `?_`.

```
?x
?хто
?назва
?_          % анонімна — кожне входження незалежне
```

### Числа

Цілі та дробні числа.

```
42
3.14
0
-1
```

### Рядки

В подвійних лапках.

```
"привіт"
"hello world"
```

### Складені терми

Функтор із аргументами в дужках.

```
батько(авраам, ісак)
вік(петро, 25)
пара(точка(1, 2), точка(3, 4))
```

Атом без дужок — це складений терм з арністю 0:

```
сонячно.     % еквівалент сонячно().
```

## Факти

Факт — це складений терм, що закінчується крапкою.

```
батько(авраам, ісак).
мати(сарра, ісак).
колір(небо, блакитний).
сонячно.
```

## Правила

Правило виражає логічний зв'язок: голова істинна, якщо тіло істинне.

### Інлайн-форма

```
правило предок(?x, ?y) якщо батько(?x, ?y).
правило дорослий(?х) якщо вік(?х, ?n) і ?n ≥ 18.
```

### Блочна форма (з відступом)

```
правило предок(?х, ?y):
    якщо батько(?х, ?z)
    і предок(?z, ?y).
```

### Кон'юнкція (і)

Цілі в тілі правила з'єднуються ключовим словом `і`:

```
правило дідусь(?x, ?z) якщо батько(?x, ?y) і батько(?y, ?z).
```

---

## Запити

Запит починається з `?-` і закінчується крапкою.

```
?- батько(?x, ісак).
?- предок(авраам, ?хто).
```

### Трасовані запити

Додайте `слід` після `?-` для виведення дерева доведення:

```
?- слід успадковує(дуб, хлорофіл).
```

Виведе дерево доведення з Unicode-символами (`├─`, `└─`, `│`).

---

## Заперечення

Заперечення через невдачу (negation-as-failure). Використовуйте `не`:

```
правило літає(?x) якщо є(?x, птах) і не не_літає(?x).
```

`не ціль` — успішне, якщо `ціль` не має жодного розв'язку.

---

## Порівняння

Арифметичні порівняння між термами:

| Оператор | Значення |
|----------|----------|
| `≥` | більше або дорівнює |
| `≤` | менше або дорівнює |
| `>` | більше |
| `<` | менше |
| `=` | дорівнює |
| `≠` | не дорівнює |

```
правило дорослий(?х) якщо вік(?х, ?n) і ?n ≥ 18.
```

Обидва операнди мають бути числовими (або змінними, що зв'язані з числами).

## Команди CLI

```bash
minilog run <файл.ml>          # Завантажити файл і виконати всі запити
minilog repl [файл.ml]         # Інтерактивна оболонка (з попереднім завантаженням файлу)
minilog check <файл.ml>        # Перевірка синтаксису без виконання
minilog version                # Показати версію
```

## Команди REPL

В інтерактивній оболонці команди починаються з `:`.

| Команда | Опис |
|---------|------|
| `:help` | Показати список команд |
| `:quit` або `:q` | Вийти |
| `:load <файл>` | Завантажити `.ml` файл |
| `:stats` | Показати статистику бази знань з розбивкою по предикатах |
| `:list <функтор>/N` | Показати факти та правила для предиката |
| `:evolve <N>` | Виконати N поколінь продукційних правил |
| `:saturate` | Пряме виведення до фіксованої точки |
| `:trace on\|off` | Увімкнути/вимкнути автоматичне трасування |
| `:clear` | Очистити базу знань |

Будь-який рядок, що не починається з `:`, інтерпретується як факт, правило або запит.

## Приклади

minilog містить 9 вбудованих прикладів в директорії `examples/`.

### `family.ml` — Родинне дерево

Біблійні патріархи. Демонструє уніфікацію зі змінними та рекурсію.

```
батько(авраам, ісак).
правило предок(?x, ?y) якщо батько(?x, ?y).
правило предок(?x, ?z) якщо батько(?x, ?y) і предок(?y, ?z).
?- предок(авраам, ?хто).
```

### `directions.ml` — Географія міст

Українські міста. Демонструє симетрію та транзитивні відношення.

### `dwh_dependencies.ml` — Залежності сховища даних

Таблиці, процедури, зв'язки читання/запису. Демонструє виявлення циклів та конфліктів запису.

### `biology.ml` — Таксономія та успадкування

Таксономія та успадкування властивостей. Демонструє рекурсію, заперечення, глибоке дерево доведення.

```
?- літає(?x).          % горобець, орел (пінгвін виключений)
?- слід успадковує(дуб, хлорофіл).   % дерево доведення
```

### `recipes.ml` — Страви та інгредієнти

Страви, інгредієнти, категорії. Демонструє багатошарове заперечення.

```
?- підходить_вегетаріанцю(?x).
?- веганська(?x).
```

### `mythology.ml` — Слов'янські боги

Боги, домени, вшанування. Демонструє моделювання культурних структур.

### `causality.ml` — Причинно-наслідкові зв'язки

Погода, транспорт. Демонструє транзитивну каузацію та спільну причину.

### `terra_tacita.ml` — Мови та комунікація

Міжвидова комунікація через мови та артефакти-перекладачі. Навчальний приклад із заповнювачами імен.

### `biology_evolution.ml` — Еволюція

Продукційні правила: мутація, видоутворення. Демонструє, що формальні правила працюють на нових сутностях без змін.

## Повідомлення про помилки

Повідомлення — англійською, з позицією (рядок/колонка) де можливо:

```
LexError at line 5, col 12: unexpected character '@'
ParseError at line 8: expected 'якщо' after rule head, got IDENT('і')
SolveError: maximum recursion depth 100 exceeded
EvaluatorError: cannot compare non-numeric terms
```

### Ієрархія помилок

- `MinilogError` — базовий клас
  - `LexError` — помилка лексичного аналізу
  - `ParseError` — синтаксична помилка
  - `SolveError` — помилка під час виведення
  - `EvaluatorError` — помилка арифметичної оцінки

## Свідомі обмеження

minilog Phase 1 навмисно не підтримує шість речей, які є у повному Пролозі. Цей розділ пояснює кожне з них: що це в принципі, як виглядає на практиці, і чому свідомо не реалізовано в minilog.

Усі шість обмежень об'єднує одна принципова ідея: **minilog Phase 1 має показати, як працюють формальні системи, а не бути робочою Пролог-системою.** Для цього достатньо уніфікації, правил, backtracking, negation-as-failure, і proof trees. Усе інше — це інженерні зручності, які не додають нічого до розуміння формальних систем, але додають значний обсяг коду, який треба писати, тестувати і пояснювати.

### 1. Диз'юнкція (`;`)

**Що це.** У стандартному Пролозі диз'юнкція — це логічне "АБО" всередині тіла правила. Замість того щоб писати два окремих правила з однаковою головою, можна записати альтернативи в одному правилі через крапку з комою:

```prolog
% З диз'юнкцією (стандартний Пролог):
батько_або_мати(X, Y) :- батько(X, Y) ; мати(X, Y).

% Без диз'юнкції (minilog):
правило батько_або_мати(?x, ?y) якщо батько(?x, ?y).
правило батько_або_мати(?x, ?y) якщо мати(?x, ?y).
```

**Чому не в minilog.** Два правила з однаковою головою роблять **точно те саме**, що диз'юнкція — движок при backtracking однаково спробує обидва варіанти. Диз'юнкція — це просто синтаксичний цукор. У навчальному прототипі цей цукор заплутує: учень починає думати, що `;` — це окрема операція, якою движок обробляє якось інакше, а насправді це лише скорочення. Краще, щоб учень бачив явні альтернативи як явні правила — це робить backtracking зрозумілим візуально.

### 2. Відсікання (`!`, "cut")

**Що це.** Cut — це спеціальний оператор Прологу, який **заморожує рішення, прийняті до цього місця в правилі**. Коли движок досягає `!`, він зобов'язується:

- Не пробувати інші альтернативи поточного правила
- Не пробувати інші правила з тією самою головою
- Не відкочувати вибір, зроблений для цілей ліворуч від `!`

Класичний приклад — "якщо-то-інакше":

```prolog
max(X, Y, X) :- X >= Y, !.
max(_, Y, Y).
```

Без `!` движок після успіху першого правила все одно перевірив би друге при backtracking і видав би `max(5, 3, 3)` як додатковий хибний розв'язок. Cut це блокує.

**Чому не в minilog.** Cut — це найменш декларативна і найбільш суперечлива частина Прологу. Є дві форми cut ("green cut" — оптимізація, "red cut" — змінює семантику), вони різняться тонко, і їхня поведінка залежить від того, як движок реалізує backtracking. Для навчального прототипу cut був би чорною магією, яка псує чисту модель "правила — це набір альтернатив". Дидактична ціна висока, користь мала. Для 9 прикладів minilog cut просто не потрібен: там, де в традиційному Пролозі був би cut, minilog використовує `не` (negation-as-failure) або додаткові умови в тілі.

### 3. Інфіксні оператори крім порівняння

**Що це.** У стандартному Пролозі можна оголосити свої інфіксні оператори через `op/3`, і потім писати вирази у природному синтаксисі:

```prolog
:- op(700, xfx, є).
:- op(600, xfy, і).

сократ є людина.
людина є смертна.
правило ?x є смертна :- ?x є людина, людина є смертна.
```

Тут `є` і `і` — користувацькі інфіксні оператори з заданим пріоритетом і асоціативністю. Це дозволяє писати код, що читається майже як англійська (або українська) проза.

У minilog інфіксними є **лише оператори порівняння**: `≥`, `≤`, `>`, `<`, `=`, `≠`. Все інше пишеться у префіксному вигляді з дужками:

```
% minilog:
є_типом(сократ, людина).
успадковує(сократ, смертність).

% А не так:
% сократ є_типом людина.
```

**Чому не в minilog.** Користувацькі інфіксні оператори вимагають окремого pass у парсері (або operator precedence parsing з таблицею пріоритетів), директив для оголошення нових операторів, і обробки неоднозначностей, коли один і той самий символ може бути префіксним, постфіксним або інфіксним. Це значне ускладнення парсера, а виграш — косметичний. Префіксна нотація `батько(авраам, ісак)` читається чудово і знімає будь-яку неоднозначність.

### 4. Списки як first-class терми

**Що це.** Один з найпотужніших інструментів Прологу — **вбудовані списки** з синтаксисом `[1, 2, 3]`, `[H|T]` (голова-хвіст), `[A, B | Rest]`, та повна бібліотека операцій: `length/2`, `append/3`, `member/2`, `reverse/2`, `nth0/3`, `msort/2`, `maplist/3` і десятки інших.

```prolog
% Класичний приклад — append:
append([], L, L).
append([H|T], L, [H|R]) :- append(T, L, R).

?- append([1,2], [3,4], X).
X = [1, 2, 3, 4].
```

Голова-хвіст нотація `[H|T]` дозволяє рекурсивно обробляти список через уніфікацію: `H` уніфікується з першим елементом, `T` — з рештою. Це елегантно і є одним з найкращих моментів Прологу.

**Чому не в minilog.** Списки — це насправді **цукор над compound-термами**. У Пролозі `[1, 2, 3]` внутрішньо представлений як `.(1, .(2, .(3, [])))` — вкладений compound з функтором `.` (крапка). Щоб списки працювали, треба додати синтаксис `[...]` у лексер і парсер, синтаксис `[H|T]` для pattern matching, реалізувати стандартну бібліотеку (append, member, length, reverse, map...), і додати семантику порожнього списку `[]`. Це значна частина обсягу Phase 1.

А головне — **жоден з 9 прикладів Phase 1 не потребує списків**. Родинні відносини, біологічна таксономія, DWH-залежності, causality — всі вони природно моделюються через окремі факти, не через списки. Якщо приклад хоче сказати "Авраам має синів Ісака і Ізмаїла", він каже це двома фактами `син(авраам, ісак).` `син(авраам, ізмаїл).`, а не одним `сини(авраам, [ісак, ізмаїл]).`

Списки знадобляться, якщо minilog колись додасть агрегування (`findall/3`, `bagof/3`, `setof/3`), бо результат агрегування — це природно список. Phase 5 (terra-tacita-canon) майже напевно їх захоче — для зберігання кількох мов одного персонажа чи кількох артефактів.

### 5. Вбудовані предикати крім порівняння та арифметики

**Що це.** Стандартний Пролог має велику бібліотеку built-in предикатів, які не визначаються через правила, а зашиті в движок на рівні C або Python. Приклади:

- **I/O:** `write/1`, `nl/0`, `read/1`
- **Метапредикати:** `call/1`, `findall/3`, `bagof/3`, `setof/3`, `forall/2`
- **Перевірка типу:** `atom/1`, `number/1`, `var/1`, `nonvar/1`, `ground/1`
- **Маніпуляція термами:** `functor/3`, `arg/3`, `=../2` (univ), `copy_term/2`
- **Динамічна база:** `assert/1`, `retract/1`, `asserta/1`, `assertz/1`
- **Управління потоком:** `\+/1` (not), `->/2` (if-then), `catch/3`, `throw/1`
- **Робота з атомами:** `atom_length/2`, `atom_concat/3`, `atom_chars/2`

(Цифра після слеша — це **арність**, тобто кількість аргументів. `findall/3` приймає 3 аргументи. У Пролозі предикати ідентифікуються парою `(ім'я, арність)`, тож `батько/1` і `батько/2` — це **різні** предикати.)

У minilog із цього всього є **тільки**:

- Порівняння: `≥`, `≤`, `>`, `<`, `=`, `≠`
- Арифметика: `+`, `-`, `*`, `/` у виразах порівняння
- `не` як negation-as-failure

**Чому не в minilog.** Built-in предикатів у стандартному Пролозі — десятки, і кожен має свою семантику, свої граничні випадки, свої тести. Це був би окремий великий модуль. А головне — **вони не потрібні для ілюстрації логіки**. Дев'ять прикладів minilog мають показати, як працює уніфікація, recursion, backtracking і proof trees. Порівняння потрібні, бо без них не можна написати `?вік ≥ 18`. Арифметика потрібна, бо без неї не можна порівняти вік. Все інше — вже інструменти для реальних програм, а не для розуміння формальних систем.

### 6. Обробка рядків

**Що це.** Рядки (`"hello world"`) у minilog **існують як терми** — їх можна зберігати у факті, можна з ними уніфікуватися, можна їх надрукувати. Але жодних **операцій над рядками** немає: ні конкатенації, ні довжини, ні пошуку підрядка, ні порівняння за алфавітом, ні case conversion, ні split.

```
% Це працює — рядок як терм:
підпис(марко, "Капітан пошти Сивої Долини").

?- підпис(марко, ?x).
?x = "Капітан пошти Сивої Долини".

% А ось це — НЕ працює:
% ?- підпис(марко, ?s) і довжина_рядка(?s, ?n).
% (предиката довжина_рядка не існує)
```

**Чому не в minilog.** У Пролозі є окрема велика тема — як взагалі представляти рядки. Існує три конкуруючі підходи: як атом (`'hello world'`), як список кодів символів (`[104, 101, 108, 108, 111]`), або як окремий тип (SWI-Prolog його ввів). Кожен зі своїми операціями і несумісностями. Для навчального прототипу занурюватися у цю плутанину — марна трата уваги.

У minilog рядки потрібні лише щоб зберігати текстові мітки — імена розділів, цитати, описи, відображувані імена, коментарі до фактів. Для цього достатньо "терм, який можна передати і надрукувати". Жодні 9 прикладів Phase 1 не вимагають ні конкатенації, ні довжини, ні будь-чого складнішого за збереження і друк.

### Загальна логіка цих обмежень

Кожне з шести обмежень зняти **легко технічно**. Додати диз'юнкцію — ~30 рядків у парсері і `solve`. Додати списки — ~100 рядків у лексері/парсері плюс невелика бібліотека. Додати cut — ~50 рядків у engine з новим сигналом у генераторі. Жодне з цих обмежень не є фундаментальним — вони всі свідомі, не технічні.

Коли і якщо знадобиться — знімати їх по одному, під конкретну задачу. Наприклад, Phase 5 (terra-tacita-canon) майже напевно захоче списки і операції над рядками (для валідації рукопису). Phase 4 (migvisor-axioms) може захотіти `findall/3` для агрегування статистики по DWH-графу. Тоді ці обмеження знімуться — під конкретну, реальну потребу. Але додавати їх **зараз**, коли задачі немає — означає ускладнювати навчальний прототип без користі.
