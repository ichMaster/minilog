## Вступ

AST walker (abstract syntax tree walker) — це набір правил, що обходять дерево розбору і обчислюють значення у кореневому вузлі. Це класичний патерн з компіляторів та інтерпретаторів: парсер будує дерево, evaluator обходить його рекурсивно.

У minilog цей патерн виражається природно: дерево — це вкладені compound terms, а правила `обчислити/2` виконують pattern matching по формі вузла і рекурсивно обходять піддерева. Це єдиний приклад у minilog, що демонструє **структурну рекурсію на складених термах** — всі інші приклади працюють з плоскими фактами.

**Для кого:** Для тих, хто хоче зрозуміти, як логічне програмування пов'язане з реальними компіляторами та інтерпретаторами.

**Пререквізити:** Phase 9 арифметика та MINILOG-061 (is/2 binding). Див. розділ "Арифметика" в [language-reference.md](language-reference.md).

## Дерево розбору як дані

Приклад використовує три типи вузлів, усі — compound terms:

| Вузол | Опис | Приклад |
|-------|------|---------|
| `число(N)` | Листок із числовою константою | `число(42)` |
| `бін(Op, L, R)` | Бінарний оператор | `бін(плюс, число(1), число(2))` |
| `функція(Name, Arg)` | Унарна функція | `функція(sqrt, число(16))` |
| `функція_pow(B, E)` | Степінь (двоаргументна) | `функція_pow(число(3), число(2))` |

Дерево для виразу `1 + 2 * 3`:

```
бін(плюс,
    число(1),
    бін(множення,
        число(2),
        число(3)))
```

Пріоритет операцій закодовано у **структурі дерева**, а не в правилах. Множення `2 * 3` є піддеревом правого аргументу додавання — тому воно обчислюється першим. Це той самий принцип, що й у реальних компіляторах: парсер вирішує пріоритет, evaluator просто обходить дерево.

### Чому compound terms підходять для дерев

У класичному Prolog дерева зазвичай представляють через списки `[op, left, right]`. У minilog без списків compound terms — природна альтернатива:

- `бін(плюс, L, R)` — це терм з функтором `бін` і трьома аргументами
- Уніфікація `бін(плюс, ?l, ?r)` витягує оператор і піддерева за один крок
- Вкладеність довільна: `бін(плюс, число(1), бін(множення, число(2), число(3)))` — це двохрівневе дерево

Це навіть читабельніше, ніж списки, бо кожен вузол має явну мітку типу.

## Структурна рекурсія

Предикат `обчислити/2` має одне правило на кожну форму вузла:

```
% Листок: значення числа — саме число
правило обчислити(число(?n), ?v) якщо ?v = ?n.

% Бінарний плюс
правило обчислити(бін(плюс, ?l, ?r), ?v) якщо обчислити(?l, ?lv) і обчислити(?r, ?rv) і ?v = ?lv + ?rv.

% Бінарний мінус
правило обчислити(бін(мінус, ?l, ?r), ?v) якщо обчислити(?l, ?lv) і обчислити(?r, ?rv) і ?v = ?lv - ?rv.
```

### Покрокове обчислення ex3 (1 + 2 * 3 = 7)

Дерево: `бін(плюс, число(1), бін(множення, число(2), число(3)))`

1. Уніфікація з правилом `бін(плюс, ?l, ?r)`:
   - `?l = число(1)`, `?r = бін(множення, число(2), число(3))`

2. Рекурсія на `?l`: `обчислити(число(1), ?lv)`
   - Уніфікація з `число(?n)`: `?n = 1`, `?lv = 1`

3. Рекурсія на `?r`: `обчислити(бін(множення, число(2), число(3)), ?rv)`
   - Уніфікація з `бін(множення, ?l2, ?r2)`: `?l2 = число(2)`, `?r2 = число(3)`
   - Рекурсія: `обчислити(число(2), ?lv2)` → `?lv2 = 2`
   - Рекурсія: `обчислити(число(3), ?rv2)` → `?rv2 = 3`
   - `?rv = 2 * 3 = 6`

4. `?v = 1 + 6 = 7`

Proof tree (через `?- слід обчислити(...)`) показує цей ланцюг візуально з box-drawing символами.

## Phase 9 арифметика в дії

Ключовий рядок у кожному правилі — це `?v = ?lv + ?rv`. Тут працює is/2 binding (MINILOG-061):

1. `?lv` і `?rv` вже зв'язані з числами (результати рекурсії)
2. `?lv + ?rv` — compound term `Compound("+", (Num(1), Num(6)))`
3. `=` бачить що `?v` незв'язана, а права сторона обчислювана
4. Evaluator рекурсивно обчислює `1 + 6 = 7`
5. `?v` зв'язується з `Num(7)`

Без Phase 9 (`+`, `-`, `*`, `/` в evaluator) і без MINILOG-061 (is/2 binding) цей приклад неможливий.

## Розділення синтаксису і семантики

Дерево розбору — це **дані**. Правила `обчислити/2` — це **інтерпретація**. Це той самий принцип, що й у реальних компіляторах:

- Парсер будує AST (синтаксис)
- Evaluator обходить AST (семантика)
- Якщо змінити evaluator, синтаксис залишається тим самим

Наприклад, якщо б замість числового обчислення ми хотіли **символьне диференціювання**, ми б написали інший предикат `диференціювати/3` над тими самими деревами `бін(...)` і `число(...)`. Дані не змінюються — змінюється лише інтерпретація. Це декомпозиція, яку використовують усі серйозні компілятори.

## Граничні випадки

### Ділення на нуль (ex8)

```
вираз(ex8, бін(ділення, число(1), число(0))).
```

Правило `обчислити(бін(ділення, ?l, ?r), ?v)` спробує обчислити `?v = 1 / 0`. Evaluator Phase 9 піднімає `EvaluatorError: division by zero`. Це правильна поведінка — помилка виникає на рівні арифметики, не на рівні логіки.

### Піфагорова відстань (ex6)

```
вираз(ex6, функція(sqrt, бін(плюс, функція_pow(число(3), число(2)), функція_pow(число(4), число(2))))).
```

Це `sqrt(3² + 4²) = sqrt(9 + 16) = sqrt(25) = 5.0`. Демонструє що вкладені функції та арифметика коректно композуються через кілька рівнів рекурсії.

### Від'ємні проміжні результати (ex7)

```
вираз(ex7, функція(abs, бін(мінус, число(3), число(10)))).
```

`abs(3 - 10) = abs(-7) = 7`. Від'ємний проміжний результат `-7` коректно обробляється функцією `abs`.

## Структурні vs обчислювальні запити

Приклад демонструє два типи запитів:

**Обчислювальні** — запускають рекурсію `обчислити/2`:
```
?- обчислити_вираз(ex3, ?v).   % → ?v = 7
```

**Структурні** — перевіряють форму дерева без обчислення:
```
?- вираз(?name, число(?n)).           % знайти всі листки
?- вираз(?name, бін(?op, ?l, ?r)).    % знайти всі бінарні вузли
```

Структурні запити використовують **тільки уніфікацію** — evaluator не викликається. Це безпечно навіть для ex8 (ділення на нуль), бо запит `вираз(ex8, бін(?op, ?l, ?r))` просто перевіряє форму, не обчислюючи значення.

## Обмеження прикладу

1. **Немає парсера.** Дерево пишеться вручну як compound term. minilog не перетворює рядок `"1+2*3"` у дерево. Для цього потрібен парсер токенів, що вимагає списків або спеціальної індексації

2. **Немає змінних у виразах.** Кожен `число(N)` містить конкретну константу. Символьні вирази зі змінними (наприклад, `x + 1`) потребують окремого механізму підстановки

3. **Немає пріоритету операцій.** Пріоритет вирішується користувачем при побудові дерева, а не evaluator-ом. У реальних компіляторах пріоритет обробляється парсером
